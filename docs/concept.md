# 컨셉 — 왜 필요한가

## 자동화의 구조적 한계

| 자동화 수단 | 잘 하는 것 | 못 하는 것 |
|---|---|---|
| 레드스톤 | 신호 기반 반복 작업 | 판단이 필요한 작업 |
| Create | 기계적 가공/운송 | 비정형 작업 (건축, 정리) |
| 주민 거래 | 고정 거래 | 새 주민 데려오기, 직업 배정 |
| AE2 | 저장/자동 크래프팅 | 네트워크 확장, 물리적 배선 |
| 호퍼/파이프 | 아이템 이동 | 컨테이너 간 판단 기반 분배 |

**공통 한계: "사람이 직접 해야 하는 잡일"이 항상 남는다.**

Agent는 이 빈틈에 들어간다. 자동화 시스템의 **글루 코드** 역할.

## 시나리오 예시

```
플레이어: "/agent Create 믹서에 재료 떨어졌어. 창고에서 철 64개 가져다 넣어"
Agent:    창고 위치 확인 → 이동 → 상자 열기 → 철 64개 꺼내기 →
          믹서로 이동 → 컨테이너에 넣기 → "완료했습니다"

플레이어: "/agent 저 언덕 평탄화해줘 10x10"
Agent:    대상 영역 스캔 → 높은 블록 제거 → 낮은 곳 채우기 → "완료"

플레이어: "/agent 주민 한 명 데려와서 대장장이로 만들어"
Agent:    주민 탐색 → 유인 (길 찾기) → 지정 위치로 이동 →
          용광로 배치 → 직업 전환 확인 → "대장장이 배정 완료"

플레이어: "/agent AE2 네트워크 채널 부족해. 컨트롤러에서 동쪽으로 케이블 연장해"
Agent:    컨트롤러 위치 확인 → AE2 API로 네트워크 상태 조회 →
          케이블 배치 → 채널 여유 확인 → "8채널 확보했습니다"
```
